<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI args → main() | --vendor, --input, --output-dir, --batch-dir supplied by user |
| main() → _run_single → create_spec_folder | vendor and spec tokens flow into path construction |
| _run_single → filesystem | Two rmtree+mkdir calls via create_spec_folder for SPLIT and READY |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04 | Tampering | create_spec_folder call sites in _run_single | mitigate | `vendor` parameter is validated by `_get_adapter()` against `VENDOR_REGISTRY` before _run_single is ever called — unknown vendor raises ValueError and exits 1 before any folder is created. `spec` = `input_path.stem` where `input_path` is resolved to an absolute path by `_resolve_path()` before entry to _run_single; `.stem` extracts only the filename stem (no directory separators). Path traversal via vendor or spec is not possible. |
| T-07-05 | Tampering | D-05 boundary — AUDIT/ not written by main.py | mitigate | Both create_spec_folder calls in _run_single use string literals "SPLIT" and "READY" as the bucket argument. No code path in main.py passes "AUDIT" to create_spec_folder. The constraint is structural — AUDIT is not a string that appears anywhere in main.py after this plan. Confirmed by acceptance criterion `grep "AUDIT" main.py` returning no match (note: "AUDIT" does not appear in comments either since no AUDIT bucket is created in Phase 7). |
| T-07-06 | Spoofing | branded workbook bytes via source_filename | accept | source_filename=input_path.name is kept unchanged (D-07). The branded workbook content is determined by the input, not by the output path. Renaming only the destination path cannot corrupt the bytes. Risk accepted — this is an intentional design constraint. |
| T-07-07 | Denial of Service | rmtree wipes stale SPLIT or READY spec dir | accept | Wipe-first is the required behavior (D-04, LAYOUT-03). The scope is a single spec directory inside a single bucket. An operator with access to the output_root can cause stale data loss by re-running; this is expected behavior, not a vulnerability. Accept. |
</threat_model>
