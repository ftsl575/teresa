# Codebase Concerns (observed state)

**Analysis Date:** 2026-05-04

*Ниже — зафиксированное состояние и расхождения источников истины. Без плана исправлений (по запросу к карте).*

## Documentation vs code

**Корневой `README.md`:**
- Содержимое фактически сводится к заголовку `# teresa` — не описывает проект.

**`spec_classifier/CURRENT_STATE.md`:**
- Живой журнал состояния с датами аудитов; версия в шапке файла (напр. 1.4.0) может не совпадать с восприятием «текущего» цикла в git/тегах — трактовать как снимок документа, не как semver процесса.

**`spec_classifier/docs/` и `spec_classifier/docs/product/TECHNICAL_OVERVIEW.md`:**
- Полезны, но отдельные фразы могут отставать (например формулировки про охват вендоров «planned» при уже подключённых адаптерах в `main.py`). **Ориентир для фактов о поведении — код.**

**`CLAUDE.md` (в т.ч. под `spec_classifier/`):**
- Богатый контекст для AI; отдельные числовые сведения (число тестов, состав golden) могут устаревать относительно текущего дерева. На дату карты: собранных тестов **775**, golden присутствуют для Lenovo/xFusion/Huawei среди прочих (см. `spec_classifier/golden/`).

**Корневой `LAUNCHER_README.md`:**
- Секция про добавление xFusion («перенести из `VENDORS_DISABLED`») не совпадает с текущим `teresa_gui.py`, где `xfusion` уже в `VENDORS_ACTIVE` (стр. ~38). Перечень кнопок GUI — по коду `teresa_gui.py`; полный CLI-список вендоров — `run.ps1` (`$ALL_VENDORS`).

## Repository / process signals

- Нет каталога `.github/workflows/` — признаков подключённого CI в репо на дату карты нет.
- Заявленные пользователем метки: много тегов без release notes, отсутствие issues/PR — процесс «solo + локально».

## Architectural / maintenance hotspots (descriptive)

**`batch_audit.py`:**
- Крупный монолитный модуль с множеством эвристик, алиасов и проверок E1–E18; связан с таксономией и YAML через загрузку карт. Изменения здесь часто затрагивают согласованность с `classifier.py` и правилами.

**Разделение парсеров:**
- Общий `src/core/parser.py` описан в обсуждениях как имеющий Dell-ориентированную историю; вендоры используют локальные `parser.py`. При правках важно не смешивать ответственность.

**Выходы аудита и источник строк:**
- Аудит исторически опирается на чтение Excel-результатов (pandas/openpyxl), а не только на `classification.jsonl` — это свойство текущей реализации, а не документа.

## Runtime boundaries

**`spec_classifier/prompts/`:**
- Не импортируется пайплайном; это инструмент разработки. Отсутствие файлов промптов не ломает `main.py`.

**AI аудит:**
- Зависит от ключа и сети; отключение через `-NoAi` оставляет детерминированную часть проверок.

## Test count drift

- Внешнее утверждение «684 теста» не совпало с фактом на машине анализа: `775 tests collected`. Использовать `pytest --collect-only` для актуального числа.
